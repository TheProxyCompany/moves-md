# Move Protocol 2026-05-02

Status: draft.

This document is the source of truth for the public Move Protocol draft.

The current Proxy implementation is older than this draft in two visible places:

- Grand Central currently stores creator provenance as `agent_id`; the public protocol field is `source`.
- Grand Central currently emits local `MoveCreated` and `MoveResolved` UI events; the public protocol Event is the external wake signal described here.

Those are implementation gaps, not separate protocols.

## Loop

```text
Agents -> Move -> Human -> Event -> Listener -> Agents -> Move
```

A Move is a durable artifact asking for human judgment.

An Event is a wake signal that something happened to a Move.

A Listener receives Events, recognizes Sources it understands, fetches Moves, and continues work.

The loop can repeat. A resolved Move can produce work, that work can produce another Move, and the next human action can wake the next Listener.

## Objects

The public protocol has four concepts:

- `Move`
- `Event`
- `Listener`
- `Source`

Proxy-specific entries, parties, timelines, execution logs, graph links, and local UI models are not part of the public protocol.

## Move

A Move object must include:

- `id`
- `source`
- `status`
- `title`
- `lede`
- `context`
- `blocks`
- `created_at`

A Move object may include:

- `made_by`
- `made_at`
- `response`
- `resolve_label`
- extension fields

Field meanings:

| Field | Meaning |
| --- | --- |
| `id` | Stable Move id. |
| `source` | Opaque return address supplied by the Move creator. |
| `status` | Current Move state. Receivers must tolerate unknown values. |
| `title` | Short title for the human-facing decision. |
| `lede` | One or two sentence summary of what is being asked. |
| `context` | Background needed to act without reading the whole originating transcript. |
| `blocks` | Ordered array of typed blocks. See [Block Types](block-types.md). |
| `created_at` | ISO 8601 creation timestamp. |
| `made_by` | Actor that acted on the Move. |
| `made_at` | ISO 8601 timestamp for the human action. |
| `response` | Structured response data keyed by block id, when the Move includes interactive blocks. |
| `resolve_label` | Optional display label for the primary resolution control. |

The current Grand Central Move model has these fields:

```text
id
agent_id
title
lede
context
blocks
status
created_at
made_by
made_at
response
on_resolve
resolve_label
```

For the public protocol, `source` is the external provenance/routing field. `agent_id` and `on_resolve` are implementation details.

Example:

```json
{
  "id": "move_123",
  "source": "proxy.party:party_abc",
  "status": "pending",
  "title": "Approve the benchmark bundle",
  "lede": "The agents prepared the comparison and need your call.",
  "context": "The bundle includes raw logs, a summary table, and the proposed follow-up.",
  "created_at": "2026-05-02T18:00:00Z",
  "blocks": [
    {
      "id": "summary",
      "type": "text",
      "content": "The benchmark run completed. The remaining question is whether this bundle is ready to send."
    },
    {
      "id": "decision",
      "type": "choice",
      "question": "What should happen next?",
      "options": ["Send it", "Revise it", "Hold"]
    }
  ]
}
```

## Status

The protocol defines these status values:

| Status | Meaning |
| --- | --- |
| `pending` | Waiting for human action. |
| `active` | The human made the Move and follow-up work should continue. |
| `completed` | The Move and its resulting work are complete. |
| `rejected` | The human declined the Move. |
| `later` | The human deferred the Move. |
| `failed` | The Move failed before or after human action. |

Receivers must tolerate unknown status values.

Current Proxy maps `made` and `done` to `completed` in Swift UI code. Grand Central currently defaults `proxy_make_move` to `active`.

## Event

An Event object must include:

- `type`
- `move_id`
- `source`

An Event object may include:

- `id`
- `created_at`

The Event is a wake signal only. It must not carry the response, resolution object, action label, block results, callback payload, transcript, or execution plan. A Listener must fetch the Move before acting.

Example:

```json
{
  "type": "move.resolved",
  "move_id": "move_123",
  "source": "proxy.party:party_abc",
  "created_at": "2026-05-02T18:04:00Z"
}
```

Initial Event types:

| Event type | Meaning |
| --- | --- |
| `move.created` | A Move was created or proposed. |
| `move.updated` | A Move changed while still unresolved. |
| `move.resolved` | A human acted on the Move. Fetch the Move for current state. |
| `move.deleted` | A Move was deleted or withdrawn. |

Receivers must ignore unknown Event types.

## Source

`source` is an opaque return address.

The protocol does not define a global source enum. A Listener may parse a Source only when it owns or recognizes that format. If the Listener does not recognize the Source, it ignores the Event.

Illustrative examples:

```text
proxy.party:party_abc
openai.codex:thread_123
anthropic.claude-code:session_abc
```

These are examples, not registered values.

## Listener

A Listener follows this algorithm:

```text
on Event:
  if source is not recognized:
    ignore
  fetch Move by move_id
  continue work from the fetched Move state
```

The Listener decides what "continue work" means. It may resume a thread, wake an agent runtime, start a job, notify another system, or produce a new Move.

The protocol does not require the agent runtime itself to stay alive. A Listener can be an always-on process, a hosted service, a webhook target, a local daemon, or another user's proxy.

## Transport

The protocol is transport independent. HTTP is the reference transport.

Reference HTTP endpoints:

```http
GET /.well-known/moves
GET /moves/:id
GET /events/stream
```

`GET /events/stream` should use Server-Sent Events when live delivery is supported:

```text
event: move.resolved
data: {"type":"move.resolved","move_id":"move_123","source":"proxy.party:party_abc"}
```

Webhook delivery and cursor-based replay are compatible transports. Polling is only a fallback.

## Discovery

Servers may expose `/.well-known/moves` so Listeners can discover protocol version and endpoint paths.

Example:

```json
{
  "protocol": "moves",
  "current_version": "2026-05-02",
  "versions": ["2026-05-02"],
  "endpoints": {
    "moves": "/moves",
    "events": "/events",
    "event_stream": "/events/stream"
  },
  "capabilities": {
    "events.sse": true,
    "events.replay": false,
    "moves.create": true,
    "moves.resolve": true
  }
}
```

## Compatibility

Receivers must ignore unknown fields.

Receivers must ignore unknown Event types.

Listeners must fetch the Move before acting on an Event.

Listeners must ignore unrecognized Sources.

Unknown block types must degrade gracefully. Renderers may show an unsupported-block fallback instead of rejecting the whole Move.

Protocol UI labels are not protocol actions. A button label such as "Make it so" is display state, not an Event payload requirement.

## Non-Goals

The Move Protocol does not define:

- an agent runtime
- a task execution engine
- a party or conversation schema
- an audit log format
- a global source registry
- a multiplayer identity system
- Proxy's internal entry schema

Those can be built on top of Moves. They are not the Move Protocol.
