# Move Protocol 2026-05-02

Status: draft.

This document is the source of truth for the public Move Protocol draft.

A Move is a durable artifact asking for human judgment.

An Event is a wake signal that something happened to a Move.

A Listener receives Events, recognizes Sources it understands, and decides how to continue.

## Objects

The public protocol has four concepts:

- `Move`
- `Event`
- `Listener`
- `Source`

## Move

A Move object must include:

- `id`
- `source`
- `status`
- `title`
- `lede`
- `context`
- `blocks`
- `created_at`

A Move object may include:

- `made_by`
- `made_at`
- `response`
- extension fields

Field meanings:

| Field | Meaning |
| --- | --- |
| `id` | Stable Move id. |
| `source` | Opaque return address supplied by the Move creator. |
| `status` | Current Move state. This draft does not define status values. |
| `title` | Short title for the human-facing decision. |
| `lede` | One or two sentence summary of what is being asked. |
| `context` | Background needed to act without reading the whole originating transcript. |
| `blocks` | Ordered array of typed blocks. See [Block Types](block-types.md). |
| `created_at` | ISO 8601 creation timestamp. |
| `made_by` | Actor that acted on the Move. |
| `made_at` | ISO 8601 timestamp for the human action. |
| `response` | Structured response data keyed by block id, when the Move includes interactive blocks. |

Example:

```json
{
  "id": "move_123",
  "source": "source_abc",
  "status": "proposed",
  "title": "Choose the invoice follow-up",
  "lede": "Acme is 14 days overdue and has not replied to the first reminder. The next message should either preserve the relationship or apply firmer pressure.",
  "context": "Acme usually pays within a week. The account is worth $48k annually, and their champion is on parental leave until June. Finance wants the invoice cleared this month.",
  "created_at": "2026-05-02T18:00:00Z",
  "blocks": [
    {
      "id": "recommendation",
      "type": "text",
      "content": "Recommended path: send a relationship-preserving reminder to the finance alias, CC the interim sponsor, and set a Friday follow-up."
    },
    {
      "id": "decision",
      "type": "choice",
      "question": "What should happen next?",
      "options": ["Send the softer reminder", "Send the firmer reminder", "Hold until Monday"]
    }
  ]
}
```

## Event

An Event object must include:

- `type`
- `move_id`
- `source`

An Event object may include:

- `id`
- `created_at`

The Event is a wake signal only. It must not carry the response, resolution object, action label, block results, callback payload, transcript, or execution plan.

Example:

```json
{
  "type": "move.made",
  "move_id": "move_123",
  "source": "source_abc",
  "created_at": "2026-05-02T18:04:00Z"
}
```

Initial Event types:

| Event type | Meaning |
| --- | --- |
| `move.created` | A Move was created. |
| `move.updated` | A Move changed. |
| `move.made` | A human made the Move. |
| `move.deleted` | A Move was deleted. |

Receivers must ignore unknown Event types.

## Source

`source` is an opaque return address.

The protocol does not define a global source enum. A Listener may parse a Source only when it owns or recognizes that format. If the Listener does not recognize the Source, it ignores the Event.

## Listener

A Listener follows this algorithm:

```text
on Event:
  if source is not recognized:
    ignore
  decide whether and how to continue
```

The Listener decides what "continue" means. It may read the Move, resume a thread, wake an agent runtime, start a job, notify another system, or produce a new Move.

The protocol does not require the originating runtime to stay alive. A Listener can be an always-on process, a hosted service, a webhook target, or a local daemon.

## Transport

The protocol is transport independent. HTTP is the reference transport.

Reference HTTP endpoints:

```http
GET /.well-known/moves
GET /moves/:id
GET /events/stream
```

`GET /events/stream` should use Server-Sent Events when live delivery is supported:

```text
event: move.made
data: {"type":"move.made","move_id":"move_123","source":"source_abc"}
```

Webhook delivery and cursor-based replay are compatible transports. Polling is only a fallback.

## Discovery

Servers may expose `/.well-known/moves` so Listeners can discover protocol version and endpoint paths.

Example:

```json
{
  "protocol": "moves",
  "current_version": "2026-05-02",
  "versions": ["2026-05-02"],
  "endpoints": {
    "moves": "/moves",
    "events": "/events",
    "event_stream": "/events/stream"
  },
  "capabilities": {
    "events.sse": true,
    "events.replay": false,
    "moves.create": true,
    "moves.made": true
  }
}
```

## Compatibility

Receivers must ignore unknown fields.

Receivers must ignore unknown Event types.

Listeners must ignore unrecognized Sources.

Unknown block types must degrade gracefully. Renderers may show an unsupported-block fallback instead of rejecting the whole Move.
